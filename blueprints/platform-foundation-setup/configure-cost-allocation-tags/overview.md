このステップでは、追加のコスト配分タグを設定します。追加するディメンションを定義し、タグ別の支出を分析するためのコストレポートビューを設定します。潜在的なコスト配分タグには次のものが含まれます:

* COST_CENTER - 会計コストセンターコード
* OWNER - 担当チームまたは個人
* PROJECT - 特定のプロジェクトまたはイニシアチブ
* APPLICATION - アプリケーションまたはシステム名


## なぜこれが重要か？

タグフレームワークを拡張することで以下が可能になります:

* 会計へのチャージバックのための**コストセンターマッピング**
* 説明責任のための**所有権追跡**
* イニシアチブベースのコスト追跡のための**プロジェクトレベルの可視性**
* 支出パターンを分析するための**コストレポートビュー**

## アカウントコンテキスト

このステップは組織アカウント（作成済みの場合）またはプライマリアカウントで実行してください。

## 前提条件

* 会計システムからのコストセンターコード
* リソースの所有権と責任の割り当て

## 主要な概念

**Snowflake のタグとは？**

[**タグ**](https://docs.snowflake.com/en/user-guide/object-tagging/introduction)は別の Snowflake オブジェクトに割り当てることができるスキーマレベルのオブジェクトです。タグはキーと値のペアとして保存され、タグ名がキーとなり、タグをオブジェクトに割り当てるときに文字列値を関連付けます。

**タグはどのように Snowflake のコスト追跡を可能にするか？**

コスト配分のために作成された[タグ](https://docs.snowflake.com/en/user-guide/object-tagging/introduction)は、ウェアハウス、データベース、その他のオブジェクトに適用でき、Snowflake の[ACCOUNT\_USAGE ビュー](https://docs.snowflake.com/en/sql-reference/account-usage)（例: WAREHOUSE\_METERING\_HISTORY、TAG\_REFERENCES）と結合して、タグ値別のクレジット消費を計算できます。これにより、コストセンター、チーム、またはプロジェクト別に支出を集計するコスト配分レポートを作成できます。詳細な例については[タグを使用したコストの帰属](https://docs.snowflake.com/en/user-guide/cost-attributing)を参照してください。

## ベストプラクティス

* **早期に始める:** オブジェクトを最初に設定するときにタグを実装するのが大幅に簡単です。既存のオブジェクトにはいつでもタグを追加できますが、Snowflake の使用ビューの過去の消費データはタグのコンテキストなしに記録されます — 新しく適用されたタグと過去のクレジット使用量を遡って関連付けることはできません。これは、コスト配分レポートがタグが適用された時点以降のタグ付き使用量のみを反映することを意味します。

* **シンプルに始める:** [1 つのオブジェクトに最大 50 のタグ](https://docs.snowflake.com/user-guide/object-tagging/introduction#tag-quotas)を設定できます（テーブルとビューを含む）。必須のディメンションから始め、FinOps の成熟度が上がるにつれて拡張します。後でいつでもタグを追加できます。

* **タグ適用を自動化する:** SQL スクリプトまたは IaC ツールを使用してタグ適用を自動化し、手動エラーを減らします。Snowflake はタグなしでリソースを作成することを自動的に防ぎません。このステップでは必須タグが欠けているリソースを監視・特定するための untagged_warehouses ビューを作成します。

* **最低限、本番環境で強制する:** 最低限、コストの可視性が最も重要な本番環境のリソースにタグを必須にすることを検討してください。

* **オブジェクトに直接コスト配分タグを適用する:** Snowflake はデータガバナンスにタグの継承を使用しますが（例: [マスキングポリシー](https://docs.snowflake.com/en/user-guide/tag-based-masking-policies)）、これはコスト配分には**適用されません**。コスト配分タグは追跡したいリソース（ウェアハウス、データベース）に直接適用する必要があります。

### 設定の質問

#### どの追加タグディメンションを使用しますか？（`additional_tag_dimensions`: multi-select）
**何を聞いているか？** 前のステップで作成されたコアプラットフォームタグ（DOMAIN、ENVIRONMENT、DATAPRODUCT、WORKLOAD、ZONE、DATA_CLASSIFICATION）以外に作成する追加タグを選択します。

**なぜ重要か？** 組織によって必要なコスト追跡の粒度が異なります。

**オプションの説明:**
**すべて（コストセンター、オーナー、プロジェクト、アプリケーション）:**
* cost\_center - 会計コストセンターコード
* owner - 担当チームまたは個人
* project - 特定のプロジェクトまたはイニシアチブ
* application - アプリケーションまたはシステム名
* 最適: 複雑なコスト配分を持つ大規模組織

**必須（コストセンター、オーナー）:**
* cost\_center - 会計へのチャージバック用
* owner - 説明責任用
* 最適: 標準のチャージバック要件を持つほとんどの組織

**最小限（オーナーのみ）:**
* owner - リソースの責任者を追跡
* 最適: 小規模組織または始めたばかりの組織

**推奨事項:** 必須（コストセンター、オーナー）から始めます。FinOps の実践が成熟するにつれてさらに追加します。
**オプション:**
- All (Cost Center, Owner, Project, Application)
- Essential (Cost Center, Owner)
- Minimal (Owner only)

#### これらのタグを新しいリソースに必須にしますか？（`require_cost_tags`: multi-select）
**何を聞いているか？**
新しいリソースが作成されるときにタグ付けをガバナンス要件にするかどうかを決定します。

**なぜ重要か？**
必須タグは完全なコストの可視性を確保しますが、ガバナンスのオーバーヘッドが増えます。

**強制の仕組み:**
Snowflake はタグなしでリソースを作成することを自動的に防ぎません。強制は次の方法で達成されます:
- **監視**: このステップでは必須タグが欠けているリソースを特定するための `untagged_warehouses` ビューを作成します
- **ガバナンスプロセス**: プロビジョニング手順にタグ付け要件を文書化する
- **ロールベースのアクセス**: ウェアハウス/データベースを作成できる人を制限し、それらのチームにタグ付け要件をトレーニングする
- **定期的な監査**: 定期的に `untagged_warehouses` ビューを確認してリメディエートする

**オプションの説明:**

**はい（すべてのリソース）:**
- ガバナンスポリシー: すべての新しいウェアハウスとデータベースにタグを付ける必要がある
- 完全なコストの可視性を確保する
- コンプライアンス監視に `untagged_warehouses` ビューを使用する

**はい（本番環境のみ）:**
- ガバナンスポリシー: 本番環境のリソースにはタグが必要
- 開発/テストリソースはオプションのタグ付け
- 可視性と柔軟性のバランスをとる

**いいえ（オプション）:**
- すべてのリソースでタグはオプション
- 最大の柔軟性
- 不完全なコスト追跡のリスク

**推奨事項:**
本番環境のコストの可視性を確保しながら開発の柔軟性を許容するために「はい（本番環境のみ）」を使用します。選択に関わらず、コストの可視性を維持するために `untagged_warehouses` ビューを定期的に確認します。
**オプション:**
- Yes (all resources)
- Yes (production only)
- No (optional)

**例:** PLAT\_INFRA
