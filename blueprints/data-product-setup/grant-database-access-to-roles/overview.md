このステップでは、カスタムロールおよびデータ製品へのアクセスが必要な追加ロールにデータベースアクセスを付与します。ステップ 3.3 でコアロール階層が確立されましたが、このステップでは以下に焦点を当てます:

1. **カスタムロールへのウェアハウスアクセスの付与** — カスタムロールはクエリを実行するためにウェアハウスアクセスが必要
2. **データ製品間のアクセス** — 他のデータ製品のロールにアクセスを付与
3. **サービスアカウントアクセス** — アプリケーションサービスアカウントのアクセスを設定

このステップは、コアロール（READ、WRITE、CREATE、ADMIN）のみを使用する場合はオプションです。

**アカウントコンテキスト:** RBAC ロールを使用してターゲットアカウントでこの SQL を実行します。

## なぜこれが重要か？

適切なデータベースアクセス付与により以下が保証されます:
- **クエリ実行**: ロールがウェアハウスアクセスを使用してクエリを実際に実行できる
- **製品間アクセス**: データ製品間のデータ共有を可能にする
- **サービス統合**: アプリケーションが必要なデータにアクセスできる

## 前提条件

- カスタム読み取りロールの作成済み（ステップ 4.1、該当する場合）
- ウェアハウスアクセスロールの作成済み（ステップ 3.2）

## 主要な概念

**アクセスコンポーネント**
完全なアクセス付与には以下の両方が必要です:
1. **データアクセス** — データベースロール付与（SC_R、DB_R など）
2. **コンピューティングアクセス** — ウェアハウス使用権付与

**付与フロー**
```
ユーザー → アカウントロール → ウェアハウスアクセスロール → ウェアハウス
                    ↓
              データベースロール → スキーマ → オブジェクト
```

**データ製品間のアクセス**
別のデータ製品のロールにアクセスを付与する場合:
```sql
GRANT DATABASE ROLE <database>.DB_R 
  TO ROLE <other_prefix>_READ;
```

**追加情報:**
* [GRANT DATABASE ROLE](https://docs.snowflake.com/en/sql-reference/sql/grant-database-role)
* [アクセス制御の権限](https://docs.snowflake.com/en/user-guide/security-access-control-privileges)


### 設定の質問

#### このデータ製品の名前は何ですか？（`data_product_name`: text）
データ製品の説明的な名前を提供します。

#### どのアカウント戦略を実装しますか？（`account_strategy`: multi-select）
**オプション:**
- Single Account
- Multi-Account (Environment-based)
- Multi-Account (Domain-based)
- Multi-Account (Domain + Environment)

#### このデータ製品はどのアカウントにデプロイされますか？（`target_account_name`: text）
Snowflake アカウントの正確な名前を入力します。

#### このデータ製品が属するドメインはどれですか？（`data_product_domain`: multi-select）
プラットフォームファウンデーションで定義されたビジネスドメインを選択します。

#### このデータ製品がデプロイされる環境はどれですか？（`data_product_environment`: multi-select）
デプロイメントの SDLC 環境を選択します。

#### きめ細かいアクセス制御のためのカスタム読み取りロールを定義してください。（`custom_read_roles`: object-list）
特定のスキーマへの限定的な読み取りアクセスが必要な外部パートナー、特定のチーム、またはアプリケーション向けのカスタムロールを定義します。

#### データ製品のウェアハウスを定義してください。（`warehouse_configuration`: object-list）
コンピューティングウェアハウスを定義します（名前、サイズ、最小/最大クラスター、自動一時停止）。
